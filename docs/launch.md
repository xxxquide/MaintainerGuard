# Launch materials

MaintainerGuard is an evidence-first GitHub Action and CLI that helps
maintainers generate merge, issue, and release readiness reports.

Use this page as reusable copy for an ethical public launch. Keep claims
specific, verifiable, and human-in-the-loop.

## Positioning

MaintainerGuard helps maintainers turn changed files, scanner outputs,
documentation and test impact, dependency signals, and repository policies into
one evidence-backed readiness report.

It is useful without AI, safe by default, and designed to reduce review noise
without replacing maintainers.

## What not to say

Do not say MaintainerGuard:

- finds every vulnerability;
- replaces human review;
- acts as an autonomous maintainer;
- guarantees security;
- is trusted by thousands;
- is used by major projects unless that is publicly true and verifiable.

## LinkedIn post

I built MaintainerGuard, an open-source GitHub Action and CLI for
evidence-first merge, issue, and release readiness reports.

The goal is simple: help maintainers understand what changed, what scanners
reported, what policy checks apply, and what should be verified before merging
or shipping. MaintainerGuard is dry-run by default, AI is optional and disabled
by default, and the report keeps major claims tied to evidence.

It does not replace human review or guarantee security. It is meant to make
maintainer review more focused by turning review signals into a concise report
and checklist.

## Reddit or Hacker News post

I built MaintainerGuard, an open-source GitHub Action and CLI that generates
evidence-backed merge, issue, and release readiness reports.

It looks at supplied PR metadata, changed files, scanner outputs, docs/test
impact, dependency signals, and repository policy rules, then produces a report
for maintainers. It is dry-run by default and does not auto-merge or claim to
find every vulnerability.

I am looking for maintainer feedback:

- Which checks would help before merging a PR?
- Which checks would be too noisy?
- Which scanner outputs should be supported first?
- What should be configurable per repository?
